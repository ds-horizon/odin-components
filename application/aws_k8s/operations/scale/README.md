| `replicas` | integer | **Yes**  | Target number of pod replicas to scale the application to. This operation changes the replica count without deploying new code or changing configuration. Use for responding to traffic changes, handling load spikes, capacity planning, cost optimization, or testing application behavior at different scales. Scaling up increases capacity and availability but increases costs; scaling down reduces costs but reduces capacity and may impact availability. Kubernetes gradually adds (scale up) or removes (scale down) pods to reach target count. During scale operations, traffic is redistributed across new replica count. Scaling is faster than redeployment (no image pull or initialization beyond standard startup). **Production:** Scale based on traffic patterns and capacity needs. For scale up: ensure cluster has sufficient resources for new pods. Monitor during scale to verify pods become healthy and traffic distributes evenly. For scale down: verify remaining capacity can handle current load with safety margin. Use minimum 2 replicas for availability (survive pod failure), 3+ for critical services (survive node failure). For sudden traffic: scale up proactively before load hits. For cost optimization: scale down during low-traffic periods but maintain minimum for HA. Consider HPA (Horizontal Pod Autoscaler) for automatic scaling based on metrics. Monitor pod resource usage to determine optimal replica count.
